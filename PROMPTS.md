

Разработать локально-развёртываемый веб-сервис для приёма и обработки заявок в ремонтную службу с двумя ролями пользователей: диспетчер и мастер. Проект должен быть готов к запуску на Windows 10 через Docker Compose, использовать SQLite для хранения данных и демонстрировать корректную обработку конкурентных запросов.

====================================
Что нужно сделать:
- собрать небольшое веб-приложение для приёма и обработки заявок в ремонтную службу.
- формат сдачи: ссылка на Git-репозиторий (GitHub), скриншоты трёх страниц, ссылка на деплой.
- проект должен запускаться локально (предпочтительно, с применение Apache24, если языки бекенда это требуют). 

====================================
Функционал (обязательный минимум)

------------------------------ 1
Роли (упрощённо): 
  диспетчер и мастер.

------------------------------ 2
Авторизация: 
  выбор пользователя на странице или логин по имени/паролю (сиды в БД).

------------------------------ 3
Заявка (Request) должна иметь поля:

`clientName` (обязательно)
`phone` (обязательно)
`address` (обязательно)
`problemText` (обязательно)
`status` (одно из): `new | assigned | in_progress | done | canceled`
`assignedTo` (мастер, может быть пустым)
`createdAt`, `updatedAt`

------------------------------ 4
Страницы/экраны (обязательные)

- Создание заявки
```
Форма создания заявки (клиент/телефон/адрес/описание). После создания заявка имеет статус `new`.
```

- Панель диспетчера
```
* список заявок
* фильтр по статусу
* назначить мастера (статус `assigned`)
* отменить заявку (статус `canceled`)
```

- Панель мастера
```
* список заявок, назначенных на текущего мастера
* действие “Взять в работу” (перевод `assigned → in_progress`)
* действие “Завершить” (перевод `in_progress → done`)
```

==============================
Действие “Взять в работу” (страница "Панель мастера") должно быть безопасным при параллельных запросах (обязательное условие): если два запроса пришли одновременно, заявка не должна “сломаться”.

Корректное поведение: один запрос успешен, второй получает отказ (например, `409 Conflict`) или понятный ответ, что заявка уже взята.

Необходимо описать в README, как это проверить (два терминала с curl или небольшой скрипт).

==============================
Требования к запуску:
- использовать Docker Compose `docker compose up`
- приложение должно быть доступно по `http://localhost:...`
- чёткая инструкция в README (composer install / migrate / seed / serve).

==============================
Для хранения данных надо использовать SQLite

==============================
Что должно быть в репозитории:
- Исходники приложения
- Миграции БД
- Сиды (минимум: 1 диспетчер, 2 мастера, несколько заявок для проверки)
- README.md: как запустить проект, тестовые пользователи, как проверить “гонку”
- DECISIONS.md (5–7 пунктов): ключевые решения
- Минимум 2 автотеста
- PROMPTS.md (обязательно)

==============================
Плюсами будут (не обязательно):
- история действий по заявке (audit log / events)
- нормальные сообщения об ошибках в UI
- аккуратная структура проекта (слои, сервисы)
- небольшой скрипт `race_test.sh` для проверки параллельного take
- размещение на сервере (деплой)
